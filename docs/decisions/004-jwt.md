# ADR 004 — JSON Web Tokens (JWT)

| Campo | Valor |
|-------|-------|
| Estado | Aceptada |
| Fecha | 2026-07-06 |
| Sprint | 2 — Authentication |

---

## Contexto

La API necesita emitir y validar tokens firmados para autenticar peticiones sin estado de sesión server-side. La spec de autenticación (v1.0.0) define dos tipos de token: access token (vida corta) y refresh token (vida larga, persistido en BD).

### Alternativas consideradas

| Opción | Descripción | Descartada porque |
|--------|-------------|-------------------|
| **JJWT** | Librería Java dedicada a JWT | — **Elegida** |
| Spring OAuth2 Resource Server | Integración Spring para JWT | Mayor complejidad para flujo custom de refresh con rotación y hash en BD |

Decisión confirmada: **D-03 — JJWT** (`jjwt-api`, `jjwt-impl`, `jjwt-jackson`).

---

## Decisión

### Librería y versión

- **JJWT 0.13.0** (compatible con Java 21).
- Artefactos Maven:
  - `io.jsonwebtoken:jjwt-api` (compile)
  - `io.jsonwebtoken:jjwt-impl` (runtime)
  - `io.jsonwebtoken:jjwt-jackson` (runtime)

### Algoritmo de firma

- **HS256** (HMAC-SHA256).
- Secreto simétrico configurado via variable de entorno `JWT_SECRET`.
- Requisito mínimo del secreto: **256 bits** (32 bytes).

### Configuración (`application.yml`)

| Propiedad | Valor | Descripción |
|-----------|-------|-------------|
| `jwt.secret` | `${JWT_SECRET}` | Secreto HMAC (obligatorio, sin default) |
| `jwt.issuer` | `lumind-intelligence-api` | Claim `iss` |
| `jwt.access-token-expiration` | `900` | Segundos (15 minutos) |
| `jwt.refresh-token-expiration` | `604800` | Segundos (7 días) |

En fases posteriores se vincularán mediante `JwtProperties` (`@ConfigurationProperties`).

### Access token

| Claim | Valor |
|-------|-------|
| `sub` | UUID del usuario |
| `email` | Email del usuario |
| `iat` | Issued at |
| `exp` | Expiración |
| `iss` | Issuer configurado |

### Refresh token

| Claim | Valor |
|-------|-------|
| `sub` | UUID del usuario |
| `type` | `"refresh"` — impide usar refresh token como access token |
| `jti` | UUID único del token |
| `iat`, `exp`, `iss` | Estándar |

### Persistencia de refresh tokens

- Tabla `refresh_tokens` en PostgreSQL (migración Flyway en Fase 2).
- Columna `token`: hash **SHA-256** en hexadecimal (64 caracteres).
- El cliente recibe el JWT en texto plano; en BD solo se almacena el hash.
- **Rotación:** cada uso válido invalida el token presentado y emite un par nuevo.
- Reutilización de token revocado → `401` + log `WARN` (posible compromiso).

### Validación del access token (filtro de seguridad — Fase 6)

1. Header `Authorization: Bearer <token>` presente.
2. Firma HMAC válida.
3. Token no expirado.
4. Claim `type` ausente o distinto de `"refresh"`.
5. Usuario existente y activo (`enabled = true`).

## Consecuencias

### Positivas

- Control explícito del flujo de emisión, validación y rotación.
- Dependencia acotada y ampliamente adoptada en el ecosistema Java.
- Separación clara entre access token (stateless) y refresh token (revocable via BD).

### Negativas

- Gestión manual de claims y expiraciones (vs. abstracción de Spring OAuth2).
- Secreto simétrico compartido: en despliegues multi-instancia todas las instancias deben usar el mismo `JWT_SECRET`.
